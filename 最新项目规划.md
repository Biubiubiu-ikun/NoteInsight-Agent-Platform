# NoteInsight Agent Platform V4.1 项目规格化

> 本版本基于 V4.0 更新：Phase 2B 已完成“视频/弹幕域 → 小红书式图文笔记域”重构。下一步不直接进入 Redis/Ranking，而是新增 **Phase 2C：账号、登录、鉴权、发帖评论闭环加固**。完成 Phase 2C 后再进入 Phase 3：缓存、热帖热评、指标和 seedgen。

---

## 0. 当前状态

已完成：

```text
Phase 1：Go backend skeleton
Phase 2：旧 video/comment/danmu API MVP
Phase 2B：Note domain pivot
  - videos → notes
  - comments → note_comments
  - 删除 danmu API
  - 新增 users / user_auth_tokens
  - 新增 note_media
  - 新增 note_likes / note_collects / note_shares / note_comment_likes
  - API 路由改为 /api/v1/notes
  - 保留 note_comments keyset pagination
  - 幂等点赞/收藏基础能力
```

现在的问题：

```text
Phase 2B 只是完成了领域模型切换和基础内容 API。
如果正式上线，还必须支持账号注册、登录、Token 鉴权、发帖、评论、点赞、收藏时从登录态获取 user_id，而不是信任请求体里的 user_id。
```

因此新增：

```text
Phase 2C：Account/Auth + Publishing Guardrails
```

## 1. 为什么 Phase 2C 必须先做

如果跳过 Phase 2C 直接做 Phase 3，会有几个问题：

```text
1. 压测请求只能传 user_id，不能模拟真实鉴权链路。
2. 发帖/评论/点赞/收藏无法校验用户身份。
3. 不能测试 banned 用户、作者权限、删除权限等真实社区规则。
4. seedgen 生成 tokens 后没有统一 auth middleware 使用。
5. 后续 Outbox、Behavior Event、RAG Query Session 都缺少可信 user_id 来源。
```

所以 Phase 2C 的目标是让项目从“帖子 CRUD 原型”升级为“可上线社区后端 MVP”。

## 3. 数据模型更新

Phase 2B 可能已经有 `users` 和 `user_auth_tokens`。Phase 2C 不建议强行推翻已有表，而是用增量方式补齐正式账号系统。

### 3.1 users 表建议字段

如果当前 users 表没有这些字段，需要补 migration：

```sql
ALTER TABLE users ADD COLUMN IF NOT EXISTS username VARCHAR(64);
ALTER TABLE users ADD COLUMN IF NOT EXISTS nickname VARCHAR(64);
ALTER TABLE users ADD COLUMN IF NOT EXISTS avatar_url VARCHAR(512);
ALTER TABLE users ADD COLUMN IF NOT EXISTS bio TEXT;
ALTER TABLE users ADD COLUMN IF NOT EXISTS role VARCHAR(32) DEFAULT 'normal';
ALTER TABLE users ADD COLUMN IF NOT EXISTS persona VARCHAR(64);
ALTER TABLE users ADD COLUMN IF NOT EXISTS status VARCHAR(32) DEFAULT 'active';
ALTER TABLE users ADD COLUMN IF NOT EXISTS updated_at TIMESTAMP DEFAULT now();

CREATE UNIQUE INDEX IF NOT EXISTS uk_users_username ON users(username);
CREATE INDEX IF NOT EXISTS idx_users_role_created ON users(role, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_users_status ON users(status);
```

role：

```text
normal
creator
brand
admin
```

status：

```text
active
banned
deleted
```

### 3.2 user_credentials

不要把密码 hash 混在大量 profile 字段里。新增 `user_credentials` 更清晰。

```sql
CREATE TABLE IF NOT EXISTS user_credentials (
    user_id BIGINT PRIMARY KEY,
    username VARCHAR(64) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    password_algo VARCHAR(32) NOT NULL DEFAULT 'bcrypt',
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_user_credentials_username ON user_credentials(username);
```

实现要求：

```text
1. 绝不存明文密码。
2. 使用 bcrypt 或 argon2id。
3. 登录时只通过 credentials 校验密码。
4. users 保存展示资料，credentials 保存登录凭据。
```

### 3.3 user_sessions

用于 refresh token 和登出。

```sql
CREATE TABLE IF NOT EXISTS user_sessions (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    refresh_token_hash VARCHAR(255) NOT NULL UNIQUE,
    user_agent TEXT,
    ip_address VARCHAR(64),
    revoked BOOLEAN DEFAULT FALSE,
    expires_at TIMESTAMP NOT NULL,
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_user_sessions_user_created ON user_sessions(user_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_user_sessions_expires ON user_sessions(expires_at);
```

### 3.4 user_auth_tokens 的定位

Phase 2B 已经有 `user_auth_tokens`。Phase 2C 后它的定位调整为：

```text
user_sessions：正式登录/刷新/登出使用。
user_auth_tokens：seedgen 和压测预生成静态 token 使用。
```

这样可以同时支持：

```text
真实用户登录链路
压测时预生成 token，避免压测时大量注册/登录干扰结果
```

## 4. Auth Token 设计

### 4.1 Access Token

```text
类型：JWT
有效期：15min - 2h，本地开发可配置为 24h
用途：API 请求鉴权
Header：Authorization: Bearer <access_token>
Claims：user_id, username, role, status, exp, iat, jti
```

### 4.2 Refresh Token

```text
类型：随机字符串
存储：只存 refresh_token_hash
有效期：7-30 天
用途：刷新 access token
登出：将 user_sessions.revoked=true
```

### 4.3 Dev / Seed Token

用于 seedgen 和 k6：

```text
Header：Authorization: Bearer <dev_token>
服务端查 user_auth_tokens.token_hash
可配置只在 ENV=local/dev/test 启用
生产环境禁用或仅内部使用
```

## 6. 内容 API 鉴权规则更新

### 6.1 公开读接口

匿名用户可以访问：

```http
GET /api/v1/notes/{note_id}
GET /api/v1/notes?category=beauty&cursor=xxx&limit=20
GET /api/v1/notes/{note_id}/comments?cursor=xxx&limit=20
```

### 6.2 登录后才能访问

```http
POST /api/v1/notes
PATCH /api/v1/notes/{note_id}
DELETE /api/v1/notes/{note_id}
POST /api/v1/notes/{note_id}/comments
DELETE /api/v1/comments/{comment_id}
POST /api/v1/notes/{note_id}/like
POST /api/v1/notes/{note_id}/collect
POST /api/v1/notes/{note_id}/share
POST /api/v1/comments/{comment_id}/like
```

### 6.3 关键规则

```text
1. 发帖 author_id 从 token 中解析，不允许请求体传 author_id 覆盖。
2. 发评论 user_id 从 token 中解析，不允许请求体传 user_id。
3. 点赞/收藏/分享 user_id 从 token 中解析。
4. 只有作者可以编辑/删除自己的帖子。
5. 只有评论作者可以删除自己的评论。
6. admin 可以删除/隐藏任意帖子和评论。
7. banned 用户不能发帖、评论、点赞、收藏、分享。
8. deleted/hidden 帖子不允许被普通用户查看或互动。
```

## 7. 中间件设计

### 7.1 AuthMiddleware

职责：

```text
1. 解析 Authorization header。
2. 支持 JWT access token。
3. dev/test 环境可支持 user_auth_tokens 静态 token。
4. 校验 token 有效性。
5. 查询或构造 AuthUser。
6. 将 user_id, role, status 写入 request context。
7. 不强制要求登录；只是“如果有 token 就解析”。
```

### 7.2 RequireAuth

职责：

```text
如果 request context 没有 user_id，返回 401。
```

### 7.3 RequireActiveUser

职责：

```text
如果 user.status != active，返回 403。
```

### 7.4 RequireOwnerOrAdmin

用于编辑/删除帖子和评论。

```text
1. service 层查询资源 owner_id。
2. 如果 current_user.role == admin，允许。
3. 如果 current_user.id == owner_id，允许。
4. 否则返回 403。
```

## 8. Service 规则

### 8.1 CreateNote

```text
Input：current_user_id + title/body/category/tags/media
Rules：
1. RequireActiveUser。
2. title/body/category 必填。
3. media 数量限制，例如 <= 9。
4. note.author_id = current_user_id。
5. status 默认 published。
6. note 和 note_media 在同一事务中写入。
```

### 8.2 UpdateNote

```text
Rules：
1. RequireActiveUser。
2. 只有作者或 admin 可更新。
3. 不允许普通用户修改 counters。
4. 更新后 updated_at 改变。
```

### 8.3 DeleteNote

建议第一版做 soft delete：

```text
UPDATE notes SET status='deleted', updated_at=now()
```

原因：

```text
1. 保留评论/互动数据方便后续 reconcile 和审计。
2. 避免物理删除导致 query_results/evidence 断引用。
```

### 8.4 CreateComment

```text
Rules：
1. RequireActiveUser。
2. note 必须存在且 status=published。
3. user_id 从 token 读取。
4. parent_id 如果不为 0，必须属于同一个 note。
5. comment_count 可先同步 +1，后续 Phase 4 改成 Outbox/最终一致。
```

### 8.5 Like/Collect

```text
Rules：
1. RequireActiveUser。
2. 使用 DB 唯一约束保证幂等。
3. 重复请求返回 200 或 409 都可以，但推荐 200 + already_liked=true，更适合前端。
4. counter 不能重复增加。
```

## 11. 给 Codex 的 Phase 2C 指令

```text
请执行 NoteInsight Agent Platform Phase 2C：账号系统、JWT 鉴权、发帖评论权限闭环。

背景：
Phase 2B 已经完成小红书式图文笔记领域重构：notes、note_media、note_comments、note_likes、note_collects、note_shares、note_comment_likes 等核心 API 已可用。现在需要把这些 API 从“请求体传 user_id 的开发模式”升级为“正式上线可用的登录态模式”。

本阶段目标：
实现正式账号注册、登录、Token 鉴权、获取当前用户、发帖/评论/点赞/收藏从 token 获取 user_id、作者权限校验、banned 用户限制。

不要做：
- Redis cache
- 热帖热评 ZSET
- RAG/Agent
- Qdrant
- Outbox/NATS
- K8s
- 复杂推荐
- 真实短信/邮箱/OAuth
- 真实图片上传

技术约束：
1. 使用 Go + Gin + sqlx + PostgreSQL。
2. 不引入 GORM。
3. handler 只做参数解析、读取 auth context 和响应。
4. 业务规则放 service。
5. SQL 放 repository。
6. 密码必须 hash 存储，禁止明文。
7. access token 使用 JWT。
8. refresh token 使用随机字符串，只存 hash。
9. 所有需要登录的接口从 JWT 中读取 user_id，不再信任请求体里的 user_id。
10. 所有幂等行为继续用数据库唯一约束兜底。

数据库迁移：
1. 如 users 缺字段，补 username、nickname、avatar_url、bio、role、status、updated_at。
2. 新增 user_credentials(user_id, username, password_hash, password_algo, created_at, updated_at)。
3. 新增 user_sessions(id, user_id, refresh_token_hash, user_agent, ip_address, revoked, expires_at, created_at, updated_at)。
4. 保留 user_auth_tokens，定位为 seedgen/k6/dev token 使用。

Auth API：
- POST /api/v1/auth/register
- POST /api/v1/auth/login
- POST /api/v1/auth/refresh
- POST /api/v1/auth/logout
- GET  /api/v1/me
- PATCH /api/v1/me

- POST /api/v1/notes 必须登录，author_id 从 token 读取。
- PATCH /api/v1/notes/{note_id} 必须登录，只有作者或 admin 可操作。
- DELETE /api/v1/notes/{note_id} 必须登录，只有作者或 admin 可操作，使用 soft delete。
- POST /api/v1/notes/{note_id}/comments 必须登录，user_id 从 token 读取。
- DELETE /api/v1/comments/{comment_id} 必须登录，只有评论作者或 admin 可操作，使用 soft delete。
- POST /api/v1/notes/{note_id}/like 必须登录，user_id 从 token 读取。
- POST /api/v1/notes/{note_id}/collect 必须登录，user_id 从 token 读取。
- POST /api/v1/notes/{note_id}/share 必须登录，user_id 从 token 读取。
- POST /api/v1/comments/{comment_id}/like 必须登录，user_id 从 token 读取。

公开读接口保持匿名可访问：
- GET /api/v1/notes/{note_id}
- GET /api/v1/notes?category=xxx&cursor=xxx&limit=20
- GET /api/v1/notes/{note_id}/comments?cursor=xxx&limit=20

Middleware：
1. AuthMiddleware：可选解析 Authorization Bearer token，有 token 则写入 auth context。
2. RequireAuth：没有登录态返回 401。
3. RequireActiveUser：banned/deleted 用户写操作返回 403。
4. RequireOwnerOrAdmin：资源作者或 admin 才能编辑/删除。

测试要求：
1. 注册成功。
2. 重复 username 注册失败。
3. 密码 hash 不等于明文。
4. 登录成功并返回 access_token、refresh_token。
5. 错误密码登录失败。
6. refresh token 可以刷新 access token。
7. logout 后 refresh token 不可再用。
8. GET /me 未登录返回 401，登录后返回当前用户。
9. 未登录创建帖子失败。
10. 登录后创建帖子成功，author_id 来自 token。
11. 创建帖子时请求体传 author_id 不生效。
12. 非作者编辑帖子失败，作者编辑成功。
13. 未登录评论失败，登录后评论成功，user_id 来自 token。
14. banned 用户不能发帖、评论、点赞、收藏。
15. 点赞帖子重复请求不重复计数。
16. 收藏帖子重复请求不重复计数。
17. 评论点赞重复请求不重复计数。
18. go test ./... 通过。

输出：
1. 目录结构变化。
2. migration SQL。
3. auth repository/service/handler/middleware 实现。
4. note/comment/like/collect/share 相关接口的鉴权改造说明。
5. curl smoke tests：register → login → create note → comment → like → collect。
6. 单测命令。
7. 本阶段不做的内容。
8. 更新后的 Phase 3 计划。
```

## 13. 给 Codex 的 Phase 3 指令

```text
请执行 NoteInsight Agent Platform Phase 3：Redis 缓存、热帖热评、指标和 seedgen MVP。

前提：
Phase 2C 已完成账号注册、登录、JWT 鉴权、发帖/评论/点赞/收藏从 token 获取 user_id。

本阶段目标：
让项目从“可上线社区后端 MVP”升级为“可压测、可观测的高并发社区后端雏形”。

具体要求：
1. 实现 note detail cache：key note:{note_id}，TTL 5-30min。
2. 实现 comments first page cache：key note:{note_id}:comments:first_page:{sort}，TTL 30-120s。
```
