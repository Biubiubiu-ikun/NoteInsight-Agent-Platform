# 1. 项目定位升级

## 1.1 原始定位

原始定位是：

> 面向中长视频内容平台的评论弹幕洞察与创作者复盘 Agent 系统。

这个定位已经比普通 CRUD 项目强，因为它把 Go 内容互动后端和 RAG/Agent 内容理解结合起来。

## 1.2 V3.0 定位

现在升级为：

> CreatorInsight Agent Platform 是一个面向中长视频内容平台的 Agent 应用平台。它提供内容互动后端、数据生成与行为模拟、Agent Runtime、工具注册、插件系统、RAG 检索服务、评测服务、可观测性和云原生部署能力，并在此平台之上落地创作者复盘、评论弹幕洞察、高能片段推荐和多垂直场景分析。

## 1.3 平台级边界

本项目可以做成偏平台级，但要明确边界。

不要做成过大的“通用 Agent 框架”或“类 Kubernetes AI 平台”。那会超出校招项目可控范围，也容易变成空架子。

正确路线是做 **应用平台级项目**：

```text
不是：单个 CreatorInsight 业务系统
而是：CreatorInsight Agent Platform

平台能力包括：
- Plugin Registry：多垂直场景插件注册和配置
- Tool Registry：Agent 工具注册和权限控制
- Agent Runtime：统一执行 planner/router/tool/synthesizer
- Retrieval Service：统一多源检索和 rerank
- Evaluation Service：统一评测 RAG/Agent 效果
- Behavior Simulator：统一生成用户、评论、弹幕、行为链
- Model Gateway：统一管理 LLM、Embedding、Reranker 调用
- Observability：统一 metrics/tracing/logging/pprof
- Deployment：统一 Docker/ECS/K8s 部署
```

一句话：

> 做一个“内容洞察 Agent 应用平台”，而不是只做一个“内容洞察应用”。

# 2. 项目核心差异化

不要把项目做成：

```text
仿 B 站
评论系统
弹幕系统
点赞系统
热榜系统
接一个 RAG 问答
```

这太普通。

要做成：

```text
Go 高并发内容互动后端
+
行为模拟与合成数据平台
+
评论弹幕智能分析
+
RAG/Agent Runtime
+
插件化垂直场景
+
在线评测系统
+
可观测性平台
+
云原生部署闭环
```

项目真正的亮点是：

1. 不是仿站，而是创作者洞察 Agent Platform。
2. 不是普通 RAG，而是字幕、评论、弹幕、外部知识库的多源时间对齐 RAG。
3. 不是简单 Tool Calling，而是有 Planner、Router、Tool Executor、Memory、Synthesizer、降级策略的 Agent Runtime。
4. 不是普通 Go CRUD，而是有热评、热榜、缓存、限流、幂等、异步事件、压测、监控和 pprof 的后端系统。
5. 不是随机造数据，而是有用户画像、行为链、时间演化、热点传播和异常行为的 Behavior Simulator。
6. 不是只做离线评测，而是有独立 Evaluation Service 管理评测集、评测任务、对比实验和报告。
7. 不是只本地跑，而是支持 Docker Compose、ECS 单机 Demo、Kubernetes/ACK 生产模拟和 HPA 实验。
8. 不是只会调用大模型，而是会做成本控制、延迟控制、降级、预计算、缓存和推理压缩。



# 2A. S级项目标准与验收边界

本项目要冲击 S 级，不靠继续堆功能，而靠让每个核心能力都有 **可运行实现、可验证指标、失败处理、监控数据和面试可解释性**。

## 2A.1 项目评级边界

```text
B 级：CRUD + Redis + 简单 RAG，能跑但同质化严重。
A- 级：Go 后端 + Redis/ZSET + RAG/Agent + 压测 + 监控 + 部署。
A 级：加入 Behavior Simulator、Evaluation Service、Agent Trace、降级和报告缓存。
S 级：加入 Evidence Store、Ground Truth、Ingestion Pipeline、Outbox/幂等、Model Gateway、权限/配额、云上验收和故障恢复。
```

S 级的核心判断标准：

```text
1. 数据闭环清楚：数据从哪里来、怎么入库、怎么进入向量索引、怎么被评测。
2. RAG 召回对象清楚：召回的不是泛泛的“文档”，而是 Evidence Units。
3. 准确率定义清楚：Retrieval、Citation、Answer、Faithfulness、Agent Tool 五层指标分开评估。
4. 生产约束清楚：超时、重试、降级、限流、成本、权限、配额、缓存、补偿都有方案。
5. 工程验收清楚：本地、ECS、K8s、压测、监控、评测报告都有验收标准。
```

## 2A.2 最小 S 级闭环

为了防止项目过大烂尾，S 级不是所有模块都做满，而是做出一个最小可展示闭环：

```text
Behavior Simulator
  → 生成视频、评论、弹幕、行为事件、标准问题、标准证据
  → PostgreSQL 主库
  → Outbox Event
  → Ingestion Worker
  → Evidence Store
  → Qdrant/BM25 Index
  → Agent Runtime 查询
  → Model Gateway 调模型
  → Agent Trace 记录执行过程
  → Evaluation Service 计算指标
  → Prometheus/Grafana/pprof 观测
  → ECS/K8s 部署和压测报告
```

只有这条链路跑通，项目才从“功能集合”变成“平台级系统”。

## 2A.3 S级验收清单

必须交付：

```text
1. 可运行：docker compose up -d 后能启动 backend、rag-agent、eval-service、postgres、redis、qdrant、nats、prometheus、grafana。
2. 可造数：seedgen 能生成压测数据和带 ground truth 的评测数据。
3. 可查询：评论、弹幕、热评、热榜接口可用。
4. 可入索引：评论主题簇、弹幕窗口、字幕片段、外部知识能进入 Evidence Store 和向量索引。
5. 可回答：Agent 能基于 evidence units 生成带 citations 的答案。
6. 可评测：eval-service 能输出 Recall@K、MRR、Citation Accuracy、Faithfulness、Tool Success Rate。
7. 可观测：Grafana 能看到 API 延迟、缓存命中率、DB 慢查询、事件积压、Agent 延迟、模型成本。
8. 可压测：k6/Locust 能复现读多写少、热门视频突发、写入突增、Agent 问答四类场景。
9. 可部署：ECS 和 K8s 至少完成一个公网演示或完整部署记录。
10. 可讲清：docs/12_system_design_evolution.md 能解释为什么这么设计，以及失败时怎么恢复。
```

## 4.1 S级端到端数据流

S 级项目必须把数据链路讲清楚。本项目的数据流分为在线交互链路、异步索引链路、Agent 推理链路和评测链路。

### 4.1.1 在线交互链路

```text
用户请求
→ Go API Gateway
→ handler 参数校验
→ service 业务规则：限流、幂等、权限
→ repository 写 PostgreSQL
→ 同事务写 outbox_events
→ 返回客户端
```

适用接口：创建评论、点赞、发送弹幕、查看视频、发起 Agent 问答。

### 4.1.2 异步索引链路

```text
outbox_events
→ outbox-publisher 发布到 NATS/Kafka
→ ingest-worker 消费事件
→ 清洗评论/弹幕/字幕
→ 聚合 comment_cluster / danmu_window / subtitle_segment
→ 写 Evidence Store
→ embedding-worker 生成向量
→ 写 Qdrant payload + vector
→ 更新索引版本
```

这个链路解决：

```text
1. 业务写入和 RAG 索引解耦。
2. 评论/弹幕高并发写入不阻塞 Agent 索引。
3. 检索数据有 evidence_id，可以被引用和评测。
4. 事件失败可以重试和补偿。
```

### 4.1.3 Agent 推理链路

```text
/api/v1/agent/chat
→ Agent Runtime
→ Planner 判断任务类型
→ Router 选择 plugin / retrieval_policy / model_policy
→ Tool Executor 调用工具
→ Retrieval Service 召回 evidence units
→ Reranker 重排
→ Model Gateway 调用 LLM
→ Synthesizer 生成 answer + citations
→ Reasoning Compression 压缩为 creator_report
→ agent_runs/tool_calls/model_invocations 全量记录
```

### 4.1.4 评测链路

```text
seedgen 生成 eval_cases + gold_evidence
→ eval-service 创建 eval_run
→ 批量调用不同策略的 RAG/Agent
→ 记录 answer/sources/latency/cost
→ 计算 Recall@K、MRR、Citation Accuracy、Faithfulness
→ 输出 rag_agent_evaluation.md
```

## 4.2 S级非目标

为了防止项目失控，明确不做：

```text
1. 不做真正多租户 SaaS 的完整计费系统，只做 workspace/project/api_key/quota 的最小实现。
2. 不做大规模推荐系统，只保留 behavior_events 作为后续扩展和压测依据。
```

# 5. 技术栈

## 5.1 Go 后端

第一版推荐：

```text
Go 1.23+
Gin
sqlx
PostgreSQL
Redis
NATS
slog 或 zap
Prometheus client_golang
OpenTelemetry Go
pprof
Docker
Kubernetes
```

原因：

- Gin 上手快，适合第一版。
- sqlx 更接近 SQL，适合面试讲索引、慢查询和事务。
- PostgreSQL JSONB 支持好，适合存 metadata、sources、report。
- Redis 用于缓存、热评、热榜、限流、幂等判重。
- NATS 比 Kafka 轻，适合本地开发；后续可以扩展 Kafka。
- pprof 和 Prometheus 能体现 Go 工程化能力。

第二版可选升级：

```text
Gin → Hertz
NATS → Kafka
单体模块化 → 微服务拆分
Docker Compose → Kubernetes / ACK
```

## 5.2 RAG / Agent

第一版推荐：

```text
Python
FastAPI
Qdrant
BM25
Embedding model
Reranker
OpenAI / Qwen / DeepSeek / Ollama 可切换
LangGraph 或自研轻量 Agent Flow
```

注意：

- 不要一开始强依赖 LangChain 模板。
- Agent Runtime 的核心设计要自己讲清楚。
- 框架只是实现工具，不是项目亮点。

## 5.3 评测与可观测性

```text
k6：单接口压测和场景压测
Locust：复杂用户行为压测
Prometheus：指标采集
Grafana：指标展示
OpenTelemetry：链路追踪
pprof：Go CPU/heap/goroutine 分析
自研 eval-service：RAG/Agent 效果评测
```

# 7. 平台核心模块设计



## 7.0 平台职责划分和实现原则

平台层不是为了显得高级，而是为了解决三个真实问题：

```text
1. 多场景：general、finance、tech_review、game_film 不能靠 if-else 硬编码。
2. 多工具：Agent 工具需要 schema、timeout、权限、启停和执行日志。
3. 多策略：检索策略、模型策略、降级策略、评测策略需要可配置、可对比。
```

实现原则：

```text
1. 控制面数据放 PostgreSQL：workspace、project、plugin、tool、policy、eval dataset。
2. 高速运行态放 Redis：缓存、限流、幂等 key、热门报告。
3. 长耗时任务走事件和 worker：索引构建、报告生成、评测运行。
4. 每次 Agent 执行必须可追踪：agent_run、tool_calls、retrieval_results、model_invocations。
5. 平台能力先做最小闭环，不做过早抽象。
```

## 7.4 Agent Runtime

Agent Runtime 不要只是“调用 LangGraph”。要能讲清楚自己的执行模型。

```text
User Question
    ↓
Planner：判断任务类型、是否需要检索、是否需要工具
    ↓
Router：选择 plugin、retrieval policy、model policy
    ↓
Tool Executor：并发或串行执行工具，带 timeout 和 retry
    ↓
Memory：读取历史问题、creator report、缓存分析结果
    ↓
Retriever：字幕/评论/弹幕/外部知识多源检索
    ↓
Reranker：重排证据
    ↓
Synthesizer：生成结构化回答和 citations
    ↓
Reasoning Compression：将长执行轨迹压缩成报告摘要
```

需要记录 agent_runs 和 tool_calls。

```sql
CREATE TABLE agent_runs (
```

## 7.6 Reasoning Compression Module

你有推理链蒸馏经验，这不是没用。它可以变成项目的 AI 工程差异化模块。

注意：这里不是输出模型隐藏推理链，而是压缩 Agent 执行轨迹，生成可审计、可缓存、可复用的结构化报告。

用途：

```text
长 Agent 执行轨迹
→ 压缩成结构化 creator insight
→ 缓存为 creator_report
→ 高并发时直接返回
→ 降低 LLM 调用次数和延迟
```

输入：

```json
{
  "question": "这个视频评论区主要争议是什么？",
  "tool_calls": [...],
  "retrieved_evidence": [...],
  "draft_answer": "..."
}
```

输出：

```json
{
  "topics": [...],
  "key_insights": [...],
  "evidence_ids": [...],
  "action_items": [...],
  "confidence": 0.82
}
```

高级目标：

- 用蒸馏的小模型或规则模型做 query routing。
- 用小模型判断是否需要完整 Agent。
- 用小模型生成报告骨架。
- 大模型只处理困难问题。

面试讲法：

> 我把推理链蒸馏经验转化成了后端系统里的“推理压缩和成本优化”模块。它不用于展示模型私有思考过程，而是把 Agent 执行轨迹压缩为结构化报告，用于缓存、降级、评测和低延迟返回。



## 7.7 Agent Trace 与调试系统

S 级 Agent 项目必须能回答：Agent 为什么这么答？工具调用哪一步慢？哪一步失败？引用证据从哪里来？

### 7.7.1 表设计

```sql
CREATE TABLE agent_trace_steps (
    id BIGINT PRIMARY KEY,
    agent_run_id BIGINT NOT NULL,
    step_no INT NOT NULL,
    step_type VARCHAR(64) NOT NULL,
    step_name VARCHAR(128),
    input JSONB,
    output JSONB,
    latency_ms INT,
    status VARCHAR(32),
    error TEXT,
    created_at TIMESTAMP NOT NULL
);

CREATE INDEX idx_agent_trace_run_step ON agent_trace_steps(agent_run_id, step_no ASC);
```

step_type 示例：

```text
planner
router
tool_call
retrieval
rerank
model_call
synthesizer
compression
fallback
```

### 7.7.2 实现方法

在 Agent Runtime 中定义统一 TraceRecorder：

```python
class TraceRecorder:
    def start_step(self, run_id, step_type, step_name, input): ...
    def finish_step(self, step_id, output, latency_ms): ...
    def fail_step(self, step_id, error, latency_ms): ...
```

每个 Agent 执行步骤必须包裹 trace：

```text
start_step
→ 执行业务逻辑
→ finish_step 或 fail_step
```

验收：

```text
GET /api/v1/agent/runs/{run_id}/trace 能返回完整执行链路。
Grafana 能看到各 step 平均耗时。
失败 case 能定位是 retrieval、tool 还是 model 失败。
```

面试讲法：

> 我没有把 Agent 做成黑盒。每次执行都会记录 planner、router、tool、retrieval、model、synthesis 的 trace，并统计每一步延迟和错误，这样才能做线上调试、性能优化和评测归因。

## 7.8 权限、配额和 API Key 最小实现

平台级项目必须有基本边界。即使不是商业 SaaS，也要避免所有用户都能无限调用 Agent。

### 7.8.1 表设计

```sql

    key_hash VARCHAR(128) NOT NULL,
    name VARCHAR(128),
    status VARCHAR(32) DEFAULT 'active',

    expired_at TIMESTAMP
);

CREATE TABLE quotas (
    id BIGINT PRIMARY KEY,

CREATE TABLE quota_usages (

    UNIQUE(project_id, quota_type, window_start)

```

quota_type：

```text
agent_requests
model_tokens
eval_runs
seedgen_jobs
```

### 7.8.2 实现方法

```text
1. API Key 只存 hash，不存明文。
2. Middleware 从 Authorization header 解析 key。
3. 查询 project_id 和 quota。
4. Redis 做实时计数，PostgreSQL 异步落库。
5. 超出 quota 返回 429，并记录 metrics。
```

Go middleware 逻辑：

```text
extract api key
→ hash
→ lookup project
→ check quota in Redis
→ set project_id into context
→ next handler
```

验收：

```text
同一个 project 超出 Agent 每分钟额度后返回 429。
Grafana 能看到 quota_rejected_total。
文档中说明这是最小权限模型，不承诺完整企业级 IAM。
```

## 7.9 Model Gateway 实际实现细化

Model Gateway 不只是封装 SDK，而是统一模型调用策略。

### 7.9.1 接口设计

```http
POST /internal/model/chat
POST /internal/model/embed
POST /internal/model/rerank
```

请求：

```json
{
  "request_id": "req_123",
  "project_id": 1,
  "purpose": "creator_report",
  "provider": "qwen",
  "model": "qwen-plus",
  "messages": [],
  "timeout_ms": 5000,
  "fallback_policy": "small_model_then_cache"
}
```

### 7.9.2 策略

```text
1. chat：支持 qwen/deepseek/openai/ollama mock。
2. embed：支持本地 embedding 或远程 embedding。
3. rerank：支持本地 reranker 或跳过 rerank。
4. timeout：每次调用必须有 context timeout。
5. retry：只对可重试错误重试，最多 1-2 次。
6. fallback：失败后返回缓存报告、结构化统计或 mock 答案。
7. cost：记录 input_tokens、output_tokens、estimated_cost。
```

### 7.9.3 验收

```text
1. 模型 provider 可以通过配置切换。
2. mock 模式可用于高并发压测，不消耗真实 token。
3. model_invocations 表有调用记录。
4. Grafana 能看到模型延迟、错误率、token 成本。
5. eval-service 能按不同 model_policy 对比效果和成本。
```

## 10.5 Outbox Pattern 实际实现

普通“写 DB 后发消息”有一个经典风险：主表写成功，但消息发布失败。为避免这个刺点，本项目采用 Outbox Pattern。

### 10.5.1 表设计

```sql
CREATE TABLE outbox_events (
    id BIGINT PRIMARY KEY,
    event_id VARCHAR(128) NOT NULL UNIQUE,
    aggregate_type VARCHAR(64) NOT NULL,
    aggregate_id BIGINT NOT NULL,
    event_type VARCHAR(128) NOT NULL,
    payload JSONB NOT NULL,
    status VARCHAR(32) DEFAULT 'pending',
    retry_count INT DEFAULT 0,
    next_retry_at TIMESTAMP,
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL
);

CREATE INDEX idx_outbox_status_retry ON outbox_events(status, next_retry_at, created_at);

CREATE TABLE processed_events (
    id BIGINT PRIMARY KEY,
    event_id VARCHAR(128) NOT NULL UNIQUE,
    consumer_name VARCHAR(128) NOT NULL,
    processed_at TIMESTAMP NOT NULL
);
```

### 10.5.2 写评论事务

```text
BEGIN;
insert comments;
insert outbox_events(event_type='comment.created');
COMMIT;
```

这样主业务数据和事件记录在同一个事务里，不会出现评论已写入但事件完全丢失。

### 10.5.3 outbox-publisher

```text
每 500ms 扫描 pending outbox_events
→ 发布到 NATS/Kafka
→ 成功后 status=sent
→ 失败后 retry_count+1，next_retry_at 指数退避
→ 超过最大重试进入 failed
```

伪代码：

```go
for {
    events := repo.LockPendingOutboxEvents(limit)
    for _, e := range events {
        err := publisher.Publish(e.EventType, e.Payload)
        if err != nil { repo.MarkRetry(e.ID, backoff(e.RetryCount)); continue }
        repo.MarkSent(e.ID)
    }
    time.Sleep(500 * time.Millisecond)
}
```

### 10.5.4 幂等消费

worker 消费前检查 processed_events：

```text
if event_id 已处理：ack and skip
else：执行业务更新 + 写 processed_events
```

同一个事件重复投递不会造成重复计数、重复索引。

### 10.5.5 补偿任务

定时 reconcile：

```text
1. 每小时重算热门视频 comment_count/danmu_count。
2. 每小时重算热榜 topN。
3. 每天扫描 failed outbox_events 并生成报告。
4. 索引缺失时重新触发 ingest。
```

面试讲法：

> 在线请求只保证主表强一致，计数、榜单、索引是最终一致。为了避免消息丢失，我使用 Outbox Pattern；为了避免重复消费，我用 processed_events 做幂等；为了处理长时间不一致，我设计了定时 reconcile。

## 10.6 哪些数据强一致，哪些最终一致

```text
强一致：评论是否创建成功、点赞是否幂等、弹幕是否写入。
最终一致：视频 comment_count、danmu_count、热榜、热评、RAG 索引、creator_report。
```

理由：

```text
1. 用户提交评论后必须能拿到 comment_id，所以评论主表强一致。
2. 点赞不能重复计数，所以 comment_likes 唯一索引强一致。
3. 热榜、计数和索引允许秒级延迟，否则高并发写入成本太高。
```

## 12.9 Ground Truth 自动生成方法


## 13.0 RAG 召回对象定义：Evidence Units

### 13.0.1 Evidence Unit 类型

## 13.0A Ingestion Pipeline 实际实现

业务数据进入 RAG 索引必须通过 Ingestion Pipeline。

### 13.0A.1 Worker 划分

```text
comment-cluster-worker：聚合评论，生成 comment_cluster。
danmu-window-worker：按时间窗口聚合弹幕，生成 danmu_window。
subtitle-worker：切分字幕，生成 subtitle_segment。

```

### 13.0A.2 评论聚类 MVP 实现

第一版不追求复杂算法，用可解释方案：

```text
1. 按 video_id 拉取最近 N 条评论。
2. 清洗文本：去重、去空、去明显刷屏。
3. 用 embedding 生成评论向量。
4. 使用 KMeans/HDBSCAN 聚类，或先用关键词规则分桶。
5. 每个簇提取 top keywords、代表性评论、情绪比例。
6. 写 evidence_units(source_type='comment_cluster')。
```

工程上为了稳定，MVP 可以先用规则 + embedding 混合：

```text
价格类关键词：价格、太贵、订阅、性价比。
隐私类关键词：隐私、数据、安全、权限。
性能类关键词：卡顿、速度、延迟、响应。
```

面试讲法：

> 第一版我用规则和 embedding 混合保证可控性，后续可以替换为更复杂聚类模型。项目重点不是炫算法，而是让聚类结果可解释、可引用、可评测。

### 13.0A.3 弹幕窗口实现

```text
1. 按 video_id + appear_second 排序。
2. 以 10s/30s/60s 为窗口聚合。
3. 计算 danmu_count、top_keywords、emotion_score、controversy_score。
4. 只把高价值窗口写入 Evidence Store，例如弹幕数 Top 20% 或情绪强度 Top 20%。
```

### 13.0A.4 索引版本管理

```text
index_version = evidence_schema_version + embedding_model_version + chunk_strategy_version
```

例如：

```text

v2_bge_m3_cluster_v2
```

每次重建索引时：

```text

4. 旧版本保留一段时间，方便回滚。
```

## 13.0B RAG 准确率定义

不要只说“准确率”。本项目分五层评估。

### 13.0B.1 Retrieval Recall@K

```text
Recall@K = TopK 中命中的 gold_evidence 数 / gold_evidence 总数
```

例子：标准证据有 4 个，Top5 命中 3 个，则 Recall@5 = 0.75。

### 13.0B.2 Retrieval Precision@K

```text
Precision@K = TopK 中相关证据数 / K
```

用于防止召回一堆无关 evidence。

### 13.0B.3 MRR

```text
MRR = 1 / 第一个相关证据排名
```

用于评估排序质量。

### 13.0B.4 Citation Accuracy

```text
Citation Accuracy = 能支撑回答结论的 citation 数 / citation 总数
```

实现方法：

```text
1. 优先基于 gold_evidence 自动判断。
2. 如果 citation evidence_id 在 gold_evidence 中，判为强正确。
3. 如果不在 gold_evidence 中，但文本语义相近，可用 LLM-as-judge 辅助判断。
4. 抽样人工复核，避免 LLM judge 自嗨。
```

### 13.0B.5 Faithfulness

```text
Faithfulness = 回答中被 evidence 支持的事实陈述数 / 回答中总事实陈述数
```

MVP 实现：

```text
1. 将 answer 拆成 claims。
2. 对每个 claim 检查 citations 是否支持。
3. 使用规则 + LLM-as-judge 打分。
4. 对低分样本人工抽查。
```

### 13.0B.6 Agent Tool Success Rate

```text
Tool Success Rate = 成功 tool_calls / 总 tool_calls
```

同时记录：

```text
tool_timeout_rate
tool_retry_count
tool_latency_p95
fallback_rate
```

# 19B. 面试防刺问答设计

## 19B.1 “你这个数据都是合成的，真实吗？”

回答：

```text
我没有把合成数据当真实业务结论，而是用于工程验证。压测数据关注规模和分布，Agent 质量数据关注 ground truth 和可评测性。生成器包含用户画像、行为链、热点传播、长尾分布和高能片段，不是简单随机字符串。报告里也明确了合成数据的边界。
```

## 19B.2 “RAG 准确率怎么定义？”

回答：

```text
我把 RAG 召回对象定义为 evidence units，包括 subtitle_segment、comment_cluster、danmu_window、external_knowledge_chunk 等。准确率不是一个指标，而是 Recall@K、Precision@K、MRR、Citation Accuracy、Faithfulness 分层评估。seedgen 会生成 gold_evidence，所以这些指标可计算。
```

## 19B.3 “消息丢了怎么办？”

回答：

```text
主表写入和 outbox_events 在同一个事务里。outbox-publisher 负责发布消息，失败会重试。消费者用 processed_events 做幂等。还有定时 reconcile 重算计数、热榜和索引，因此系统采用最终一致性但可修复。
```

## 19B.4 “Agent 很慢怎么办？”

回答：

```text
首先有 Model Gateway 统一超时、重试和 fallback。其次热门视频会预计算 creator_report。高并发时优先返回缓存报告。LLM 超时则返回结构化统计结果。Reasoning Compression 会把长执行轨迹压缩成可缓存的洞察，降低延迟和成本。
```

## 19B.5 “平台化是不是过度设计？”

回答：

```text
我只做应用平台级，不做通用 Agent 框架。平台化只解决三个实际问题：插件可扩展、工具可管理、评测可复现。Project/Plugin/Tool Registry 都是最小实现，不做复杂 SaaS 和计费系统。
```
