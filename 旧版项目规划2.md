# NoteInsight Agent Platform V4.0 项目规格化

> 面向小红书式图文笔记社区的内容互动后端、行为模拟、AI 搜索/RAG Agent、Evaluation Service 与云原生可观测闭环。

## 0. 给 Codex 的总指令

我要把当前 CreatorInsight Agent Platform 从“中长视频/弹幕场景”重构为“小红书式图文笔记社区场景”。

新项目名建议：**NoteInsight Agent Platform**。

项目目标不是做小红书 UI，也不是爬取真实小红书数据，而是构建一个用于证明 Go 后端工程能力 + AI 应用工程能力的 S 级项目：

- Go 高并发图文笔记社区后端。
- 用户、创作者、帖子、图片说明/OCR、评论、点赞、收藏、分享、关注、搜索行为。
- Redis 缓存、ZSET 热帖/热评、限流、幂等、Outbox、异步事件。
- Behavior Simulator 生成符合社区分布的用户、帖子、评论、点赞、收藏、搜索和点击行为。
- Evidence Store 将帖子正文、图片说明/OCR、评论、评论主题簇、行为摘要转为可引用证据。
- RAG/Agent 支持用户自然语言查询，召回最相关帖子和评论证据，返回回答 + related_notes + citations。
- Evaluation Service 评估 Note Recall@K、Evidence Recall@K、MRR/NDCG、Citation Accuracy、Faithfulness、延迟和成本。
- Prometheus/Grafana/pprof/k6/Locust/ECS/K8s 完成可观测性、压测和部署闭环。

Codex 每次推进必须包含：当前目标、为什么这么设计、目录结构变化、数据模型、API、代码实现、测试方式、压测方式、监控指标、可能 bug、面试追问、下一步。

## 1. 业务定位

### 1.1 新定位

NoteInsight Agent Platform 是一个面向小红书式图文笔记社区的 AI 搜索与评论洞察平台。

用户可以输入生活决策类问题，例如：

- “油皮夏天有什么平价持妆粉底推荐？”
- “杭州周末适合拍照的咖啡店有哪些？”
- “通勤电脑包 500 元以内怎么选？”
- “新手 A 醇怎么建立耐受？”

系统会检索相关帖子、图片说明/OCR、评论、评论主题簇、收藏/点赞等行为摘要，并返回：

- 综合回答。
- 相关帖子 related_notes。
- 支撑回答的 citations。
- 每篇帖子为什么相关。
- 查询会话和召回结果落库，便于复盘和评测。

### 1.2 工程定位

本项目不是创业级产品验证项目，不声称解决已被市场验证的行业级痛点。

它选取小红书式图文社区作为合理业务场景，重点证明：

1. 高并发内容互动后端能力。
2. 社区产品真实分布数据生成能力。
3. RAG/Agent 工程化能力。
4. Evaluation Service 评测能力。
5. 缓存、异步事件、幂等、限流、可观测和部署能力。

## 2. 从视频版到图文笔记版的映射

| 旧视频版 | 新图文笔记版 |
|---|---|
| videos | notes / posts |
| subtitles | note body / note sections / image captions / OCR text |
| danmus | 删除 |
| danmu_window | 删除，替换为 comment_cluster / behavior_summary |
| video hot ranking | note hot ranking |

## 5. Redis 设计

### 5.1 note detail cache

```text
key: note:{note_id}
type: STRING(JSON)
ttl: 5min - 30min
```

### 5.2 comment first page cache

```text
key: note:{note_id}:comments:first_page:{sort}
type: STRING(JSON)
ttl: 30s - 120s
```

### 5.3 hot notes ranking

```text
key: ranking:notes:daily
key: ranking:notes:{category}:daily
type: ZSET
member: note_id
score: hot_score
```

热度公式：

```text
hot_score = view_count * 1
          + like_count * 3
          + collect_count * 8
          + comment_count * 6
          + share_count * 5
          + recent_growth_score * 10
          + search_click_score * 6
          - time_decay
```

### 5.4 hot comments

```text
key: note:{note_id}:hot_comments
type: ZSET
member: comment_id
score: like_count * 5 + reply_count * 3 + freshness_score + intent_weight
```

### 5.5 query result cache

```text
key: query:{project_id}:{query_hash}:{retrieval_policy}:{index_version}
type: STRING(JSON)
ttl: 10min - 1h
```

### 7.2 RAG 检索逻辑

```text
用户问题
→ query normalization
→ query intent classification
→ metadata filter 构造

→ note-level aggregation
→ rerank notes
→ rerank evidence
→ LLM synthesis
→ 返回 answer + related_notes + citations
→ 保存 query_session 和 query_results
```

### 7.3 note-level aggregation

不能只按 evidence 排序，否则 Top20 可能全来自同一篇笔记。

```text
note_score = max_evidence_score * 0.45
           + avg_top_evidence_score * 0.20
           + collect_score * 0.15
           + comment_quality_score * 0.10
           + freshness_score * 0.10
```

## 8. Evaluation 指标 V4.0

### 8.1 Note Recall@K

```text
Note Recall@K = TopK 中命中的 gold_note_ids 数 / gold_note_ids 总数
```

### 8.2 Evidence Recall@K

```text
Evidence Recall@K = TopK evidence 中命中的 gold_evidence 数 / gold_evidence 总数
```

### 8.3 Note Precision@K

```text
Note Precision@K = TopK 相关帖子数 / K
```

### 8.4 MRR / NDCG@K

```text
MRR：第一个相关帖子或证据排得是否靠前。
NDCG@K：相关性高的帖子是否排在更前面。
```

### 8.5 Citation Accuracy

```text
Citation Accuracy = 正确支持回答结论的 citation 数 / citation 总数
```

### 8.6 Faithfulness

```text
Faithfulness = 被 evidence 支持的事实陈述数 / 回答事实陈述总数
```

### 8.7 Query-Level 工程指标

```text
RAG retrieval latency
rerank latency
LLM latency
first token latency
total latency
cost per query
query cache hit rate
fallback rate
```

### 9.3 真实分布原则

```text
1. 5%-10% 用户是发帖用户。
2. 20% 热门帖子承载 80% 访问量。
3. 20% 活跃用户产生 80% 写操作。
4. 创建时间分散在 3-12 个月。
5. 评论、点赞、收藏不能均匀分布。
6. 高收藏笔记通常具有更强“攻略/清单/教程/避雷”属性。
7. 热门笔记形成 Redis 热 key。
8. 少量 spammer 产生重复评论，用于限流和反刷测试。
```
